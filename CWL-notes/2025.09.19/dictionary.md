# A
**Active Reconnaissance** — Direct interaction with the target to gather technical details (e.g., scanning, service identification, directory enumeration).
**Android/iOS Sandbox** — OS-level isolation of apps with runtime permissions and digital signing to protect other apps’ data.
**Audit (Security Audit)** — Assessment of compliance with policies/standards and the organization’s process and technical security posture.
**AWS/GCP/Azure Review (whitebox)** — Structured review of cloud configurations and controls to identify misconfigurations and risks.

# B
**Blackbox** — Testing with only publicly available information and no provided credentials.
**Blue Team** — Defensive team responsible for monitoring, detection, and response during engagements.
**BlueKeep** — Example of a high‑profile memory‑corruption vulnerability cited in training materials.
**Brute‑force** — Repeated credential guessing attempts against exposed services.
**Bug Bounty** — Program paying researchers for responsibly reported vulnerabilities in products or services.
**Buffer Overflow** — Overwriting adjacent memory by sending oversized input to a fixed‑size buffer.

# C
**Chain of Trust** — Digital signing and verification of system components to prevent untrusted code from running. 
**CVE (Common Vulnerabilities and Exposures)** — Public catalog of disclosed vulnerabilities with standardized identifiers. 
**CVSS (Common Vulnerability Scoring System)** — Standard for rating vulnerability severity used for prioritization. 

# D
**DDoS (Distributed Denial of Service)** — Overwhelming services with traffic to degrade or disrupt availability.

# E
**Enumeration (Path Enumeration)** — Systematic discovery of directories/files on web servers to expand attack surface.
**Escalation of Privileges** — Elevating from low‑privileged accounts to administrative/root access. 
**E‑commerce (Scope Area)** — Business domain often included in web and API security testing scopes. 

# F
**Full Disclosure** — Immediate public release of vulnerability details without a remediation grace period.
**Frontend (Scope Area)** — UI layer of web applications included in web pentest coverage.

# G
**GitLeaks** — Scanner detecting hardcoded secrets and sensitive data in Git repositories.
**Graybox** — Testing with partial knowledge or limited accounts/documentation provided.
**G Suite / Google Workspace (Scope Area)** — Productivity platform included in cloud/SaaS security reviews.

# I
**Incident Analysis** — Investigation and assessment of security events and their impact.
**Insider Threat** — Malicious or accidental risk originating from within the organization.
**Integer Overflow** — Arithmetic wrap/overflow causing incorrect values and downstream logic errors.
**IoT (Internet of Things)** — Connected devices and embedded systems included as a testing area.

# J
**Jailbreak (iOS)** — Obtaining root privileges on iOS devices by exploiting security mechanisms.

# K
**Kubernetes** — Container orchestration platform included in infrastructure/cloud security reviews. 

# M
**Mapowanie aplikacji (Application Mapping)** — Systematic discovery of functions, roles, and routes to chart attack surface. 
**Mobile Security** — Android/iOS controls like sandboxing, code signing, and verified boot impacting pentest setup. 

# N
**Nessus Essential** — Entry‑level host/service vulnerability scanner with a plugin feed. 
**Network Segmentation** — Separation of network zones to limit lateral movement and exposure. 
**Nuclei** — High‑speed, template‑driven scanner focusing on web exposures and mass recon. 

# O
**Office 365 / Microsoft 365 (Scope Area)** — SaaS productivity suite commonly reviewed in cloud assessments.
**OpenVAS** — Open‑source vulnerability scanner with community feeds and compliance policies.
**OSINT (Open‑Source Intelligence)** — Passive collection from public sources to inform targeting.
**OWASP Dependency‑Check** — SCA tool detecting vulnerable third‑party libraries in projects. 

# P
**Password Spraying** — Trying common passwords against many accounts within throttling limits. 
**Pegasus** — Example tooling leveraging 0‑day chains to compromise mobile devices. 
**Pentest (Penetration Test)** — Controlled attack simulation to find and validate impactful vulnerabilities. 
**Persistence** — Maintaining controlled access post‑compromise (e.g., new users, SSH keys).
**Pivoting** — Using a compromised host to reach otherwise inaccessible internal subnets/services.
**Purple Team** — Collaborative offensive/defensive exercises to improve detection and response.

# R
**Red Team** — Adversary‑emulation team conducting realistic attack campaigns.
**Responsible Disclosure** — Coordinated vulnerability disclosure with a remediation window before publication. 
**Retest** — Follow‑up testing to confirm remediation effectiveness. 
**Rules of Engagement (RoE)** — Agreed scope, methods, contacts, and constraints governing the test. 
**Reconnaissance (Passive/Active)** — Information gathering before exploitation to map assets and attack paths. 

# S
**S‑SDLC (Secure SDLC)** — Integrating security activities into software development lifecycle. 
**Sandbox** — Isolation boundary preventing unauthorized inter‑app data access and API use.
**Scope** — Explicit definition of in‑scope assets, environments, credentials, and constraints. 
**SonarQube Free** — Community code analysis for quality and basic security hotspots. 

# W
**WAF (Web Application Firewall)** — HTTP traffic filter; testing may be performed with WAF enabled/disabled per RoE. 
**WAN/VPN/LAN (Scope Area)** — Network layers and connectivity models evaluated in infrastructure tests. 
**Whitebox** — Testing with full access to code/configuration and supporting documentation.
